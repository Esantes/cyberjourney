A focused tracker for the long climb from script-kiddie to operator.

CyberJourney is a personal cybersecurity learning tracker, built as a portfolio project. It unifies progress across OverTheWire, TryHackMe, Hack The Box, and PortSwigger Web Security Academy into one local-first React app. The goal is not a SaaS. The goal is a small, well-made codebase I'd happily put my name on and use myself.

Why feature-first

  • Locality of behaviour. Everything an integration owns — config, adapter, detail page, types — sits in one folder. Deleting a platform is rm -rf features/x.
  • Stable surfaces. ui/, lib/, and hooks/ change rarely; features/ changes constantly. Separating churn from stability keeps reviews small.
  • Test boundaries. Each feature folder may carry its own __tests__. Cross-feature tests live under test/.
SEC · 02 / 13

Component architecture

Four tiers, strict one-way dependencies. A higher tier may import from any lower tier; never the reverse.

Tier 4 · Pages
Route-level views. Compose features. No business logic, no styling that isn't layout. e.g. DashboardPage, TrackerDetailPage.
Tier 3 · Features
Domain-aware composite components and hooks. May read store, dispatch actions, call adapters. e.g. TrackerCard, ChallengeRow, SkillRadar.
Tier 2 · UI Kit
Headless-ish presentation primitives. Knows nothing about trackers. e.g. Card, Button, Badge, Tabs.
Tier 1 · Lib / Hooks
Pure functions, storage drivers, framework-agnostic utilities. Unit-testable in isolation.

Component contract

  • Every component is a named function export, never default. One component per file, file named in PascalCase.jsx.
  • Props are destructured at signature; required props first, optional second, className + ...rest last.
  • UI primitives accept className and forward it via cn(). They do not own colors as props — variants only (e.g. tone="phos" | "mag" | "muted").
  • Composite components own layout; primitives own atomic style. No magic numbers in JSX — only Tailwind classes or token-derived utility classes.
  • Side effects live in hooks, not components. A component file with fetch or localStorage in it is a smell.

Page structure & routing

React Router 6 with nested routes under a persistent AppShell. URL is the source of truth for navigation state; nothing important lives only in component state.

PathComponentPurpose
/DashboardPageHero KPIs, four tracker cards, recent activity, current streak.
/trackersTrackersPageAll four platforms side-by-side with filter + sort.
/trackers/:slugTrackerDetailPagePer-platform deep dive. Slug ∈ otw|thm|htb|psw.
/trackers/:slug/:challengeIdChallengeViewSingle challenge: notes, status, time spent, tags.
/timelineTimelinePageAppend-only activity log across platforms.
/skillsSkillsPageTag-derived skill graph and per-skill maturity.
/journalJournalPageLong-form write-ups and lessons learned.
/settingsSettingsPageTheme, motion preference, data export / import, danger zone.
*NotFoundPage404 with terminal-style fallback.

Layout strategy

  • Shell-with-outlet. AppShell renders the side rail (desktop) or bottom tab bar (mobile) plus a <Outlet/>. Page components never render chrome.
  • Suspense per route. Each lazy-loaded page is wrapped in a RouteSuspense with a skeleton matching its final layout.
  • Scroll restoration is per-route, opt-out on detail pages so back-navigation feels stateful.
SEC · 04 / 13

State management strategy

Three rings of state, smallest to widest. The rule: state lives at the narrowest ring that still works.

R1Local / Component

useState · useReducer

Form inputs, hover, open/close, transient UI. Never persisted, never serialised.

R2Cross-cut / Session

React Context

Theme, motion preference, hotkey scope, toast queue. Read-mostly, low-churn values only.

R3Domain / Persisted

Context + reducer

Tracker progress, journal entries, settings. Mediated by storageAdapter — swap driver, not store. Zustand could slot in here later; doesn't earn its keep at this size.

Store shape

useTrackerStore ├── user: { handle, joinedAt, prefs } ├── trackers: Record<slug, TrackerState> │ └── TrackerState │ ├── slug: 'otw' | 'thm' | 'htb' | 'psw' │ ├── lastSyncedAt: ISO string | null │ ├── items: Record<id, ChallengeProgress> │ └── stats: { completed, inProgress, total, timeMin } ├── journal: JournalEntry[] ├── events: ActivityEvent[] # append-only ├── settings: { theme, motion, density, scanlines } └── _meta: { schemaVersion, driver, hydratedAt }

Persistence flow

  • Selector-driven writes. Components call action creators (logCompletion, updateNote); the store mutates and the driver persists via a single persist middleware.
  • Driver abstraction. storageAdapter exposes get / set / list / remove. Today the only driver is localStorageDriver. If I ever want cloud sync, I write a new driver — the store and UI don't change.
  • Schema versioning. _meta.schemaVersion gates a tiny migration table. Bumps require a data/migrations/v00X.js file. Painless to start, painful to retrofit — so I start with it.
  • Optimistic UI. All writes apply locally first. Since the only driver is synchronous localStorage, this is trivial today; it stays trivial if a network driver ever gets added.
SEC · 05 / 13

Progress tracking architecture

Each platform is wildly different — wargames, rooms, machines, labs. We model them through a single TrackerAdapter contract so the UI is platform-agnostic.

Adapter contract
  • id
  • label
  • getCatalog() → Challenge[]
  • normalize(raw) → ChallengeProgress
  • computeStats(state) → Stats
  • icon, accent
Challenge model
  • id · platform-scoped
  • title, track
  • difficulty: 1–5
  • tags: skill[]
  • status: idle | wip | done | stuck
  • timeMin, completedAt
  • notesRef → journal id
Derived metrics
  • Completion ratio
  • Avg. time / difficulty
  • Streak (consecutive active days)
  • Skill maturity (weighted tag scores)
  • Velocity (7-day rolling)

Why an adapter, not bespoke pages

  • Uniform UI. TrackerCard, ProgressRing, ChallengeRow only know the canonical model. Adding a fifth platform means writing one adapter + one config.
  • Pure math. progress.math.js is a set of pure functions taking TrackerState. 100% testable, zero React.
  • Event sourcing-lite. Every state-changing action also appends to events with type + payload + timestamp. The timeline page reads exclusively from events.

Reusable UI system

Small kit, sharp edges, zero dependencies beyond what's in the stack. Built on Tailwind utility classes + token-driven variants. shadcn/ui is not a dependency — we own the surface.

Button

Tones: phos · ghost · mag · danger. Sizes: sm · md · lg. Loading + icon-only states.

Badge / Tag

Badge = status (filled). Tag = skill/category (outlined). Never invert.

Input · Select · Toggle

Same height (36 / 44 mobile). Always paired with FormField wrapper for label + error.

Modal · Drawer

Drawer is the mobile equivalent of Modal. Routing-aware: ?modal=… opens, back closes.

Tabs · Segmented

URL-synced tabs for detail pages; uncontrolled Segmented for in-card filters.

Toast · Empty · Skeleton

Toasts are session-only. Every list view ships an empty + skeleton state on day one.

Spacing & radii

  • 4px base scale — 2 · 4 · 8 · 12 · 16 · 20 · 24 · 32 · 48 · 64. No values off-scale.
  • Radii — 3px chips, 4px cards, 8px modals. No pill radii on rectangles.
  • Borders — 1px solid --line default; 2px --phos accent rail; never both on the same edge.
  • Elevation — flat by default. Use border, not shadow, to separate surfaces. A single soft shadow exists for floating overlays only.

Cyberpunk discipline

  • One glow per screen, maximum. Usually the active progress ring or a critical CTA.
  • No neon gradients. Flat fills only. Saturation comes from one accent against deep neutral, not from rainbows.
  • Scanlines are opt-in. Off by default; toggleable in Settings via .scanlines body class. Respects prefers-reduced-motion.
  • Monospace is structural, not decorative. It labels data; it doesn't replace prose.
  • No emoji. Status uses badges + symbols (●, ◐, ◯, ⨯).

Animation strategy

Framer Motion, used like a scalpel. Animation conveys state change or hierarchy; it never decorates idle UI.

// Route transition

Cross-fade + 6px y-shift. 220ms, easeOut. Wrapped in <AnimatePresence mode="wait"/> at the route boundary.

opacity 0→1 · y 6→0 · 220ms
// Card enter

Staggered 30ms per item, max 8 items animated; the rest snap. Only on first paint of a list — not on re-renders.

stagger(0.03) · y 8→0
// Progress ring

Spring on completion delta. Subtle scale-pulse (1 → 1.03 → 1) only when value increases.

spring · stiffness 220 · damping 22
// Modal / Drawer

Backdrop fades 140ms; surface springs 180ms. Drawer slides from bottom on mobile, right on desktop.

backdrop 140ms · surface spring
// Status pill

layoutId on the pill so transitioning wip → done morphs in place rather than re-rendering.

layoutId="status-{id}"
// Hover

Cards lift via border-color brightening only; no translateY, no shadow swap. Costs are predictable.

border-color 120ms

Rules

  • Reduced motion is law. Wrap all non-essential animation with usePrefersReducedMotion(). Reduced mode = opacity-only, no movement, no springs.
  • No infinite loops. No background ambient animations. The grid does not pulse. The cursor does not blink in chrome.
  • Durations are tokens: fast 120 · base 180 · slow 240 · modal 280. Don't hand-pick durations in components.
  • Spring presets live in lib/motion.js — spring.soft, spring.snappy. Components reference these by name.

Mobile-first strategy

Designed at 360px first, then unlocked up. Touch ergonomics, navigation, and data density are all locked at the small end before desktop adds breathing room.

Rules

  • Bottom tab bar <md, side rail ≥md. Same destinations, different chrome.
  • 44px minimum hit target for any tap surface. Buttons grow from 36 → 44 at md down.
  • Drawer instead of Modal <sm. Same component contract, different mounted surface.
  • No horizontal scrolling outside of explicit carousels. Tables collapse to stacked cards.
  • Sticky page headers are 56px tall and never overlap content; safe-area insets respected.
  • Font min size 13px, body 15px. line-height 1.5+ across the board.

Personal roadmap

No deadlines, no investors, no exit ramp. Just things I might explore next, in roughly the order I'd reach for them.

NOW · SHIPPED
Local-first app

All four trackers, Bandit 0–33 pre-seeded, markdown journal, skills, achievements, terminal, pomodoro. The MVP of me.

NEXT · QUALITY
Polish what's there

More Bandit notes seeded from my own write-ups. PWA so it installs. Keyboard shortcuts. Smaller bundle.

MAYBE
CSV / Markdown import

Drop in a folder of write-ups, parse front-matter for tags, auto-link to challenges. Still 100% local.

MAYBE
Cloud sync, only if I want it

Swap the storage driver for a network-backed one. The UI doesn't change. No multi-tenancy, no billing — just one user (me) on more than one device.

What's deliberately not on this list

  • Accounts, auth, multi-tenancy. This is one person's tracker. Adding accounts would double the scope and triple the deploy story.
  • Public profile pages, leaderboards. The whole point is honest self-tracking; the moment scores are public, I'd start gaming them.
  • Payments, subscriptions, tiers. Nothing here is for sale. Every feature ships free or doesn't ship.
  • Admin dashboards, role systems, audit logs. No users to administrate.

What I locked down anyway, because it pays back

  • Storage adapter interface. Even with only one driver today, the seam is in place. Cheap insurance.
  • Append-only events. The activity log is its own little event log. The heatmap reads from it, future features could too.
  • Pure adapters per tracker. A fifth platform is a single folder.
  • Schema versioning, day one. Trivial to start with, brutal to retrofit.

Project rules, in one breath

The full charter lives in PROJECT_RULES.md. This is the cheat-sheet.

ACoding

Strictness pays back

Named exports only. One component per file. No default exports outside pages/. Hooks own side effects, components own JSX. Zod-validate everything entering the store.

BUI consistency

Tokens or it didn't happen

No literal colors, no off-scale spacing, no bespoke radii. cn() for class merging. Variants over props. Cards never set their own width.

CState

Narrowest ring

useState before context, context before store. Store actions are verbs. Persisted writes go through the adapter, never localStorage direct.

DMotion

Restraint is a feature

One glow per screen. Animations convey change, not vibe. Reduced-motion is enforced, not optional. Use motion tokens.
